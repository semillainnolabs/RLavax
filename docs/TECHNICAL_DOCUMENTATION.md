# Technical Documentation

## Tech Stack

RapiLoans leverages a robust and modern technology stack designed for security, speed, and user experience.

### Blockchain & Smart Contracts
*   **Primary Network**: Avalanche C-Chain (Target for Mainnet) for high throughput and low finality time.
*   **Testnet**: Base Sepolia (for Proof of Concept) due to availability of Aave and Morpho testnet deployments.
*   **Smart Contract Language**: Solidity ^0.8.20.
*   **Core Protocol Integrations**:
    *   **Morpho Blue**: Provides the core lending market infrastructure, allowing for isolated lending pairs and efficient capital usage.
    *   **Aave**: Utilized as the underlying yield source for collateral.
*   **Oracles**: Custom `WaUSDCMXNBOracle` to price MXNB against the collateral.

### Frontend
*   **Framework**: Next.js 16 (React 19) with TypeScript for type safety and server-side rendering.
*   **Styling**: Tailwind CSS v4 for responsive and modern UI design.
*   **Web3 Integration**:
    *   **Ethers.js v6** & **Viem**: For blockchain interaction.
    *   **Privy**: For seamless wallet connection and user authentication (`@privy-io/react-auth`).
*   **UI Components**: `@heroicons/react` for iconography, custom components for Modals and Toasts.

## Architecture Decisions

### 1. The "Yield-for-Zero-Interest" Model
The core architectural decision is to **subsidize borrowing costs** by redirecting the yield generated from the borrower's collateral.
*   **Traditional Model**: Borrower supplies collateral -> Protocol locks it -> Borrower pays interest.
*   **RapiLoans Model**: Borrower supplies USDC -> Protocol deposits USDC into Aave -> Yield (aUSDC) accrues to the Protocol/Subsidy Pool -> Borrower pays interest on Morpho -> Protocol uses accrued yield to offset/subsidize this interest.

### 2. Morpho Blue for Lending Markets
We chose **Morpho Blue** over a traditional fork of Aave/Compound because:
*   **Permissionless Market Creation**: Allows us to create a specific `MXNB / WaUSDC` market.
*   **Efficiency**: Higher LTVs and better capital efficiency.
*   **Isolation**: Risk is isolated to the specific pair, protecting the broader protocol.

### 3. Wrapper Contract (`WaUSDC`)
To implement the subsidy logic without forking complex lending protocols, we created a **Vault/Wrapper contract** (`WaUSDC`):
*   It complies with **ERC-4626** standard.
*   It wraps `aUSDC` (Aave interest-bearing USDC).
*   It intercepts the yield generated by Aave.
*   It tracks "Virtual Shares" to ensure fair distribution of subsidies.
*   This decoupling allows the lending logic (Morpho) to remain standard while the subsidy logic lives in the collateral token itself.

## Implementation Approach

### Smart Contracts
*   **`WaUSDC.sol`**: The heart of the subsidy mechanism. Users deposit USDC in Aave, then the app deposits the `aUSDC` in WaUSDC contract. The contract mints `WaUSDC` tokens which represent the *principal* (stripping the yield). These tokens are then used as collateral.
*   **`DebtLens.sol`**: A helper contract to calculate accurate debt and interest accrued on Morpho Blue markets without needing heavy external dependencies.
*   **`MockMXNB.sol`**: Represents the stablecoin being borrowed (for the PoC).
*   **`WausdcMxnbOracle.sol`**: Provides the exchange rate between the collateral and the loan asset to ensure solvency.

### Frontend Integration
The Next.js application acts as the orchestrator:
*   **`useMorphoLoan` Hook**: Encapsulates the complex multi-step Borrowing logic (Approve -> Deposit Aave -> Approve Vault -> Deposit Vault -> Supply Morpho -> Borrow) and Repaying logic (Approve -> Calculate Accrued Interest -> Repay Morpho -> Withdraw Collateral -> Unwrap WaUSDC -> Withdraw Aave).
*   **`useMorphoLend` Hook**: Encapsulates the multi-step Lending logic (Approve -> Deposit Morpho) and Reedeming one in Morpho.
*   **Simulation**: Real-time calculation of required collateral based on LTV and Oracle price before the user submits a transaction.
*   **Step-by-Step Execution**: The UI guides the user through the multiple transactions required (6-7 steps for borrowing, and 5 for repaying) with clear status updates, abstracting the complexity of the underlying DeFi "lego" interactions.
